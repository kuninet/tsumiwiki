# 03. 非機能要件

要件IDの凡例: `NFR-xx-nn`。

## 3.1 性能

- **NFR-PERF-01** 通常規模の文書(〜100KB程度のMarkdown)の表示・編集開始は2秒以内。
- **NFR-PERF-02** 数千文書規模のライブラリでもフォルダツリー表示・検索が実用速度で動作すること。
- **NFR-PERF-03** 編集中のWYSIWYGレンダリングは入力に追従し、体感遅延がないこと(日本語IME入力を含む)。

## 3.2 可用性・データ保全

- **NFR-AVL-01** 小規模チーム利用のため高可用性構成は要求しない。ただしサーバー再起動後にデータ・ロック状態が破綻しないこと。
- **NFR-AVL-02** 文書データのバックアップは、ファイルサーバー上のGit bareリポジトリへの定期push(またはコミット都度のpush)により行う。push失敗時もローカルの動作は継続し、ログに記録する。
- **NFR-AVL-03** 保存処理はアトミックに行い、保存失敗時に文書ファイルが破損しないこと。

## 3.3 セキュリティ

- **NFR-SEC-01** 全機能はログイン必須。APIも認証なしではアクセス不可とする。
- **NFR-SEC-02** パスワードは bcrypt 等の適切なアルゴリズムでハッシュ化して保存する。
- **NFR-SEC-03** XSS対策: Markdown中のHTML/スクリプトはレンダリング時にサニタイズする。
- **NFR-SEC-04** パストラバーサル対策: ファイルパスを扱うAPIはライブラリルート外へのアクセスを禁止する。
- **NFR-SEC-05** 外部ネットワークへ公開する場合はHTTPS必須とする(初期はローカルネットワーク利用のためHTTPを許容)。

## 3.4 互換性・移植性

- **NFR-COMP-01** 保存されるMarkdownはCommonMark + GFM(表・チェックリスト等)準拠とし、Obsidian・VS Code等の一般的ツールで閲覧・編集可能なこと。
- **NFR-COMP-02** 文書のメタデータ(タグ等)はYAMLフロントマターに保存し、Obsidianと互換の形式とする。
- **NFR-COMP-03** ファイルエンコーディングはUTF-8(BOMなし)、改行はLF。
- **NFR-COMP-04** 日本語ファイル名・フォルダ名(NFC正規化)を扱えること。macOS由来のNFD問題に配慮する。
- **NFR-COMP-05** 対応ブラウザ: Chrome / Edge / Safari / Firefox の各最新版(PC/Mac)。

## 3.5 運用・保守

- **NFR-OPS-01** 初期リリースはWindows PC上で稼働させる。Windows上でのNode.js直接実行、またはDocker(Docker Desktop等)のいずれでも起動できること。将来のAWS移行を見据えコンテナ化にも対応する。
- **NFR-OPS-04** Windowsサーバー稼働に伴うパス区切り・ファイル名・改行コードの差異を吸収すること(保存されるMarkdownは常にUTF-8/LF)。
- **NFR-OPS-02** ライブラリフォルダのパス、セッションタイムアウト、ロック自動解放時間等は設定ファイルまたは環境変数で変更可能とする。
- **NFR-OPS-03** アプリケーションログ(ログイン、保存、エラー)を出力する。

## 3.6 開発・拡張性

- **NFR-DEV-01** 将来要件(複数ライブラリ、リアルタイム共同編集、AWSデプロイ)を見据え、ストレージアクセス・認証部分は差し替え可能な構造とする。
- **NFR-DEV-02** 自動テスト(最低限、保存・履歴・ロック等コア機能のAPIテスト)を整備する。
