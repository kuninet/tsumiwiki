# 05. データ・履歴要件

## 5.1 ライブラリのフォルダ構成

ライブラリは設定で指定した1つのルートフォルダ。構成例:

```
<library-root>/            ← 設定で指定。Gitリポジトリとして初期化される
├── .git/
├── .obsidian/             ← Obsidianヴォルトをコピーした場合に存在。ツリー非表示・Git管理外
├── .trash/                ← ごみ箱。削除した文書の移動先(Obsidian互換)
├── folder-a/
│   ├── 文書1.md
│   ├── 文書2.md
│   └── image-20260704.png ← 添付は文書と同じフォルダに保存(既定)
└── folder-b/
    └── 文書3.md
```

- 既存のObsidianヴォルトをそのままライブラリルートとして利用できる(FR-OBS-01)。初期化時に既存ファイルを変換・移動しない。
- **ファイル名=文書タイトル**とする(Obsidianと同じ)。本文先頭の `#` 見出しとは独立で、リネームしてもH1を自動変更しない(逆も同様)。日本語ファイル名を正式サポートする(NFC正規化・Gitの `core.quotepath=false` 等で対応。NFR-COMP-04)。wikilink(`[[文書名]]`)はファイル名で解決されるため、日本語ファイル名の維持はObsidian互換の前提条件でもある。ファイルシステム禁止文字(`/ \\ : * ? " < > |`)は全角等へ置換する。
- 添付ファイルは既定で「文書と同じフォルダ」に、衝突しないファイル名(タイムスタンプ等)で保存する(既存ヴォルトの運用に合致)。保存先は設定で変更可能。文書からは `![[ファイル名]]` または相対パスで参照する。
- `.obsidian/` 等の設定系ドットフォルダは文書ツリーに表示せず、Gitコミット対象から除外する(`.gitignore` に登録)。
- `.trash/` はごみ箱として扱う。文書の削除は `.trash/` への移動とし、ごみ箱画面から復元・完全削除できる(FR-DOC-07)。削除・復元の操作はGitコミットに記録する。

## 5.2 文書ファイル形式

- Markdown(CommonMark + GFM)、UTF-8(BOMなし)、改行LF。
- メタデータはYAMLフロントマターに保存する(Obsidian互換):

```markdown
---
tags: [設計, 議事録]
created: 2026-07-04T10:00:00+09:00
updated: 2026-07-04T12:34:56+09:00
---

# 文書タイトル

本文...
```

- フロントマターはWikiが管理するが、外部ツールで編集されても破綻しないよう寛容にパースする。既存Obsidianヴォルトのフロントマター(`tags` 等)をそのまま利用し、Wiki側で不要な書き換えを行わない。
- 内部リンクはObsidian互換のwikilink記法(`[[文書名]]`)、画像埋め込みは `![[ファイル名]]` および標準記法 `![](相対パス)` の両方に対応する(FR-OBS-02/03)。
- 編集者情報はフロントマターではなくGit履歴で管理する(ファイル内容を汚さない)。

## 5.3 履歴管理(Git)

- ライブラリルートをGitリポジトリとして管理し、以下の操作でコミットを作成する:
  - 文書の保存(明示保存時のみ。自動保存の下書きはコミットしない)
  - 文書・フォルダの作成/リネーム/移動/削除
  - 画像・添付の追加
- コミットauthorは「ログインユーザーの表示名 + システム内メールアドレス」を設定し、誰の編集かを記録する。
- コミットメッセージは操作内容を機械生成する(例: `edit: folder-a/文書1.md`)。
- 過去版の閲覧・差分・復元はGitの履歴を利用する。復元は revert 相当の新規コミットとして記録し、履歴は改変しない。
- 外部ツールによるライブラリ直接編集で未コミット変更が生じた場合は、Wikiアクセス時に自動コミットで取り込む(author はシステムユーザー)。

## 5.4 アプリケーションデータ(文書以外)

文書ライブラリとは別に、アプリ用データベース(SQLite想定)で以下を管理する:

| データ | 内容 |
|---|---|
| ユーザー | ID、表示名、パスワードハッシュ、ロール、有効/無効 |
| セッション | ログインセッション情報 |
| 編集ロック | 文書パス、ロック保持ユーザー、取得日時、最終更新日時 |
| 下書き(自動保存) | 編集中の未保存内容(保存/破棄で削除) |
| 検索インデックス | 全文検索用インデックス(再構築可能なキャッシュ扱い) |

- DBが失われても文書ライブラリ(Git)からサービスを再構築できること(ユーザー再登録を除く)。

## 5.5 データ量の想定

| 項目 | 想定 |
|---|---|
| 文書数 | 〜数千 |
| 1文書サイズ | 通常〜100KB |
| 添付含むライブラリ総容量 | 〜10GB程度 |
