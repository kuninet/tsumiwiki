# 07. 将来要件(初期リリースのスコープ外)

初期リリースには含めないが、概要で言及されており将来対応したい要件。設計時に拡張の妨げにならないよう考慮する。

## FUT-01 AWSデプロイ

- ECS/Fargate等でのコンテナ稼働、EFSまたはS3+同期でのライブラリ保持、RDSへのDB移行を想定。
- 初期からDocker化(NFR-OPS-01)しておくことで移行コストを下げる。
- 公開時はHTTPS化・WAF等のセキュリティ強化が必要。

## FUT-02 複数ライブラリ(「個人」と「共用」)

- ユーザーごとの個人ライブラリ + チーム共用ライブラリの構成。
- ライブラリ単位のアクセス権管理が必要になる。
- 初期実装ではライブラリルートをハードコードせず「ライブラリ」を抽象として扱い、複数化に備える(NFR-DEV-01)。

## FUT-03 作図機能(Draw.io的なもの)

- 有力案: draw.io(diagrams.net)の埋め込みエディタ連携。`.drawio.svg` / `.drawio.png` をライブラリ内に保存し、画像として文書へ埋め込む方式なら Markdown 互換を保てる。
- Mermaid記法のレンダリングは初期リリースに前倒し済み(FR-EDIT-10)。本項はGUI作図(Draw.io連携)のみを将来要件として残す。

## FUT-04 リアルタイム共同編集

- 編集ロック方式から、CRDT(Yjs等)ベースの同時編集への発展。
- Tiptap は Yjs 連携の公式サポートがあるため、エディタ選定はこの将来要件と整合する。

## FUT-05 モバイル対応

- スマートフォン/タブレットでの閲覧・簡易編集。初期はPC/Macのみ。

## FUT-06 高度な権限管理

- フォルダ/文書単位の閲覧・編集権限、閲覧専用ユーザー等。

## FUT-07 Dataviewクエリの実行対応

- 既存ヴォルトではDataviewを使用しているが、初期リリースでは実行せずコードブロック表示に留める(FR-OBS-08。内容は破壊しない)。
- 将来対応する場合も、まずはTABLE/LIST等のよく使う型のサブセットから検討する。dataviewjs(JavaScript実行)はセキュリティ上の考慮が必要なため慎重に判断する。
