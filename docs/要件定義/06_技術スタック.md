# 06. 技術スタック(推奨構成)

技術スタックは「おまかせ」の方針を受け、以下を推奨構成とする。実装フェーズ開始時に最終確定する。

## 6.1 推奨構成

| レイヤ | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | React + TypeScript + Vite | 情報・ライブラリが豊富。Tiptapとの組み合わせ実績が多い |
| WYSIWYGエディタ | Tiptap(ProseMirrorベース) | Markdown対応WYSIWYGのデファクト。IME(日本語入力)対応が成熟。ツールバー・ショートカット・拡張が容易 |
| Markdown変換 | remark / unified 系 | CommonMark+GFM対応。フロントマター・wikilink等の拡張が容易 |
| バックエンド | Node.js + TypeScript(Fastify または Express) | フロントと言語統一。ファイルI/O・Git操作のライブラリが揃う |
| Git操作 | isomorphic-git または simple-git | 保存ごとのコミット、履歴取得、差分、revertをアプリから制御 |
| アプリDB | SQLite(better-sqlite3) | 小規模利用に十分。運用が軽い。将来AWS移行時はRDS等へ差し替え可能な構造にする |
| 認証 | セッションCookie方式(自前実装 or Lucia等) | 小規模・独自ID/パスワード方針に合致 |
| 全文検索 | SQLite FTS5(日本語はtrigramトークナイズ) | 追加ミドルウェア不要で日本語検索に対応 |
| 図表 | Mermaid(mermaid.js) | Obsidian標準対応と互換。コードブロックをクライアント側でレンダリング |
| 配布・実行 | Windows PC上のNode.js直接実行 or Docker | 初期リリースはWindows PC稼働。将来のAWS(ECS等)はコンテナで対応 |

## 6.2 構成イメージ

```
[ブラウザ (React + Tiptap)]
        │ HTTPS/HTTP (REST API + セッションCookie)
        ▼
[Node.js APIサーバー]
   ├── 認証・セッション ──────── SQLite
   ├── ロック管理・下書き ────── SQLite
   ├── 検索インデックス ─────── SQLite (FTS5)
   └── 文書サービス ──────────  ライブラリフォルダ (Markdown + attachments)
                └── Git操作 ──── .git (履歴)
```

## 6.3 主要な技術リスクと対応

| リスク | 対応 |
|---|---|
| WYSIWYG⇔Markdown往復での差分発生(FR-EDIT-06) | Tiptap+remarkの変換を早期にプロトタイプし、対応記法の範囲を確定する |
| 日本語IMEとエディタの相性 | ProseMirror系はIME対応実績が多いが、早期に実機(macOS/Windows両方)で検証する |
| 日本語ファイル名のNFC/NFD問題(macOS) | サーバー側でNFC正規化を統一適用 |
| Git操作の性能(数千文書時) | 履歴取得は文書単位(`git log -- <path>`)に限定。必要ならキャッシュ |
| Windowsサーバー特有の差異(パス区切り・ファイルロック・改行) | パス操作はライブラリで抽象化。保存はUTF-8/LFに統一。開発時からWindowsでの動作確認を行う |
| ファイルサーバーbareリポジトリへのpush(バックアップ) | UNCパス/ネットワークドライブ経由のpushを早期検証。失敗時リトライとログ記録 |
