# 01. アーキテクチャ設計

## 1.1 全体構成

```
[ブラウザ]
  React SPA (Vite build) + Tiptapエディタ
      │  HTTP(S) — REST API (JSON) + セッションCookie
      ▼
[Windows PC上のNode.jsサーバー (Fastify)]
  ├── 認証・セッション管理 ─────────┐
  ├── 文書サービス(読み書き・ロック)  ├── SQLite (app.db)
  ├── 検索サービス(FTS5) ──────────┘
  ├── Gitサービス(コミット・履歴・push)
  │       │
  │       ▼
  ├── ライブラリフォルダ(Markdown + 添付 + .git)
  │       │  git push(バックアップ)
  │       ▼
  └── ファイルサーバー上のbareリポジトリ(\\fileserver\...\tsumiwiki.git)
```

- サーバーは単一のNode.jsプロセス。本番ではクライアントのビルド成果物(静的ファイル)も同プロセスから配信する(ポート1つで完結)。
- 文書の実体はライブラリフォルダのMarkdownファイル。SQLiteは認証・ロック・下書き・検索インデックス等の付帯情報のみ(要件 05章5.4: DB喪失時もライブラリから再構築可能)。

## 1.4 実行・配布

- **開発**: `pnpm dev` でVite devサーバー(HMR)+ APIサーバーを同時起動。ViteのproxyでAPIを中継。
- **本番(Windows PC)**: `pnpm build` 後、Node.jsで `server` を起動。クライアント静的ファイルはFastifyの静的配信で提供。Windowsサービス化にはNSSM等を利用(手順は導入ドキュメントに記載)。
- **前提ソフトウェア**: Node.js LTS、Git for Windows。
- **コンテナ**: Dockerfileを用意し、将来のAWS移行(ECS等)に備える(NFR-OPS-01)。
- **設定**: 環境変数または `config.json` で以下を指定する。

| 設定キー | 内容 | 既定値 |
|---|---|---|
| `LIBRARY_PATH` | ライブラリフォルダの絶対パス | (必須) |
| `PORT` | 待ち受けポート | 3000 |
| `DB_PATH` | SQLiteファイルパス | `<data>/app.db` |
| `SESSION_TTL_MINUTES` | セッションタイムアウト | 480 |
| `LOCK_TIMEOUT_MINUTES` | 編集ロック自動解放(FR-LOCK-03) | 30 |
| `ATTACHMENT_DIR_MODE` | 添付保存先(`same-folder` / 指定フォルダ名) | `same-folder` |
| `BACKUP_REMOTE` | バックアップ先bareリポジトリ(UNCパス可) | (任意) |
| `BACKUP_PUSH_INTERVAL_MINUTES` | push間隔 | 10 |

## 1.5 セキュリティ設計

| 項目 | 方式 |
|---|---|
| 認証 | ログインでセッションID発行。Cookie(httpOnly, SameSite=Lax, 本番HTTPSならSecure)に格納。セッションはSQLiteに保存しTTL管理(NFR-SEC-01, FR-AUTH-05) |
| パスワード | bcrypt(コスト10)でハッシュ化(NFR-SEC-02)。ポリシーは空文字禁止のみ(FR-AUTH-07) |
| CSRF | SameSite=Laxに加え、変更系APIはカスタムヘッダ `X-Requested-With` 必須とする |
| XSS | エディタ/ビューアはProseMirrorのスキーマで許可ノードのみ描画。生HTMLブロックはエスケープ表示(NFR-SEC-03)。MermaidはsecurityLevel=strictで実行 |
| パストラバーサル | APIで受けるパスは正規化(NFC・区切り統一・`..`解決)後、ライブラリルート配下であることを必ず検証する共通関数を経由(NFR-SEC-04)。`.git`・`.obsidian` 等への直接アクセスは拒否 |
| 認可 | ロールは admin / user の2種(FR-AUTH-03)。ユーザー管理・強制ロック解除・完全削除系APIはadminのみ |
| 監査 | ログイン成否・保存・削除・復元・ロック操作をアプリログに記録(NFR-OPS-03) |

## 1.6 横断的な処理方針

- **文字コード・正規化**: ファイル読み書きはUTF-8(BOMなし)/LF固定。パス・ファイル名はサーバー側で常にNFC正規化(NFR-COMP-03/04, NFR-OPS-04)。
- **アトミック書き込み**: 一時ファイルに書いてからrename(NFR-AVL-03)。
- **エラー形式**: APIエラーは `{ error: { code, message } }` に統一(03章)。
- **ログ**: pinoで構造化ログをファイル出力(ローテーションあり)。
