# 02. データモデル設計

## 2.1 データの二層構造

| 層 | 実体 | 役割 |
|---|---|---|
| 一次データ | ライブラリフォルダ(Markdown・添付・`.git`) | 文書の正本。Obsidian互換(要件05章) |
| 付帯データ | SQLite `app.db` | 認証・セッション・ロック・下書き・検索/タグのインデックス |

原則: **SQLite側は全て「ライブラリから再構築可能なキャッシュ」または「文書と独立した運用データ」**とする。文書の内容・メタデータ(タグ等)の正本は常にMarkdownファイル側にある。

## 2.2 SQLiteスキーマ

```sql
-- ユーザー(FR-AUTH-01〜04)
CREATE TABLE users (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  username      TEXT NOT NULL UNIQUE,          -- ログインID
  display_name  TEXT NOT NULL,                 -- 表示名(Gitコミットauthorに使用)
  password_hash TEXT NOT NULL,                 -- bcrypt
  role          TEXT NOT NULL DEFAULT 'user',  -- 'admin' | 'user'
  disabled      INTEGER NOT NULL DEFAULT 0,
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL
);

-- セッション(FR-AUTH-05)
CREATE TABLE sessions (
  id         TEXT PRIMARY KEY,                 -- ランダム128bit以上
  user_id    INTEGER NOT NULL REFERENCES users(id),
  created_at TEXT NOT NULL,
  expires_at TEXT NOT NULL
);

-- 編集ロック(FR-LOCK-01〜04)。文書パスに対して高々1行
CREATE TABLE locks (
  doc_path     TEXT PRIMARY KEY,               -- ライブラリルートからの相対パス(NFC)
  user_id      INTEGER NOT NULL REFERENCES users(id),
  acquired_at  TEXT NOT NULL,
  refreshed_at TEXT NOT NULL                   -- ハートビートで更新。停滞で自動解放
);

-- 自動保存の下書き(FR-EDIT-08)。ロック保持者のみ書けるため文書ごとに1行
CREATE TABLE drafts (
  doc_path   TEXT PRIMARY KEY,
  user_id    INTEGER NOT NULL REFERENCES users(id),
  content    TEXT NOT NULL,                    -- Markdown全文(フロントマター込み)
  updated_at TEXT NOT NULL
);

-- 文書インデックス(ツリー・タグ・検索のキャッシュ)
CREATE TABLE doc_index (
  doc_path   TEXT PRIMARY KEY,
  title      TEXT NOT NULL,                    -- ファイル名(拡張子なし)
  folder     TEXT NOT NULL,                    -- 親フォルダ相対パス('' = ルート)
  updated_at TEXT NOT NULL,                    -- ファイルのmtime
  size       INTEGER NOT NULL
);

-- タグインデックス(フロントマター + インラインタグ。FR-OBS-06)
CREATE TABLE doc_tags (
  doc_path TEXT NOT NULL,
  tag      TEXT NOT NULL,                      -- 正規化済み(先頭#なし。階層タグ a/b はそのまま)
  source   TEXT NOT NULL,                      -- 'frontmatter' | 'inline'
  PRIMARY KEY (doc_path, tag, source)
);
CREATE INDEX idx_doc_tags_tag ON doc_tags(tag);

-- 全文検索(FR-NAV-03)。trigramで日本語対応
CREATE VIRTUAL TABLE doc_fts USING fts5(
  doc_path UNINDEXED,
  title,
  body,
  tokenize = 'trigram'
);
```

- 日時は全てISO 8601文字列(UTC)で保存し、表示時にローカライズする。
- `doc_index` / `doc_tags` / `doc_fts` はインデックスサービスが管理する派生データ。全削除→再構築(フルリインデックス)コマンドを用意する。

## 2.3 インデックス更新方式

| タイミング | 処理 |
|---|---|
| サーバー起動時 | ライブラリを全走査し、mtime/size が `doc_index` と異なるファイルのみ再パース(差分リインデックス) |
| Wiki経由の保存・作成・削除・移動 | 該当文書のみ即時更新 |
| 外部変更の検出時(06章) | 変更ファイルのみ再パース |

- パース内容: フロントマター(`tags`・`created`・`updated`)、インラインタグ(`#タグ`)、本文プレーンテキスト(FTS用。コードブロック内の `#` はタグとみなさない)。
- インラインタグの抽出規則はObsidianに合わせる: `#` の直後に空白・記号を含まない文字列。行頭または空白の直後のみ。コードブロック・インラインコード内は除外。

## 2.4 ライブラリ側のデータ規約(再掲・確定)

要件定義05章のとおり。設計上の確定事項のみ追記する。

- 文書の識別子はライブラリルートからの**相対パス**(NFC・`/` 区切りに統一)。リネーム・移動はパス変更として扱い、ロック・下書き・インデックスの各テーブルも追随更新する。
- wikilinkの解決(`[[文書名]]` → パス)は Obsidian の「最短パス」規則に合わせ、`doc_index.title` の一致で検索する。同名文書が複数ある場合はフォルダ付き表記(`[[folder/文書名]]`)を優先し、曖昧な場合は最初の一致を採用する。
- `.gitignore`(ライブラリ内)には `.obsidian/` を登録。`.trash/` はコミット対象(削除履歴を残すため)。
- 添付の命名: `image-YYYYMMDDHHmmss.png` 形式(同名衝突時は連番付与)。保存先は既定で文書と同じフォルダ(FR-OBS-05)。

## 2.5 バックアップ・復旧の観点

- `app.db` が失われた場合: ユーザー・セッション・ロック・下書きは消えるが、文書は無傷。ユーザー再登録+フルリインデックスで復旧(要件05章5.4)。
- ライブラリが失われた場合: bareリポジトリ(バックアップ先)から `git clone` で復旧(06章)。
